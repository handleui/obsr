Detent uses Better Auth with GitHub OAuth as the API identity provider.

## Authentication flow

- Auth endpoints live under `https://observer.detent.sh/api/auth/*`.
- The browser login flow uses Better Auth generic OAuth with `provider: "github"`.
- On successful sign-in, the Better Auth session is established and can be used for API auth.
- Protected API calls can use:
  - `Authorization: Bearer <token>`
  - `X-Detent-Token` / `X-API-Key` compatibility headers (when issued).

## Via CLI (Recommended)

The CLI handles the auth flow and token lifecycle:

```bash
detent login
```

## API login flow

You can start OAuth login using Better Auth’s social sign-in API with provider `github` and then return to the callback path below:

```ts
const response = await authClient.signIn.social({
  provider: "github",
  callbackURL: "https://your-app.com/callback",
});
```

OAuth callback target:

```bash
https://observer.detent.sh/api/auth/oauth2/callback/github
```

## Using tokens

Include a session token in the `Authorization` header for requests:

```bash
curl https://observer.detent.sh/v1/projects \
  -H "Authorization: Bearer YOUR_TOKEN"
```

```json
{
  "sub": "user_01ABC123",
  "email": "user@example.com",
  "iat": 1704067200,
  "exp": 1704068100
}
```

`sub` maps to the Better Auth user ID.

## Error responses

### Invalid Token

```json
{
  "error": "Invalid or expired token"
}
```

Status: `401 Unauthorized`

### Missing Token

```json
{
  "error": "Authorization header required"
}
```

Status: `401 Unauthorized`

## Security best practices

- Store tokens securely (system keychain, encrypted storage)
- Use HTTPS for all API requests
- Prefer short-lived tokens and rotate when you need fresh auth context
- Revoke tokens when users log out
- See Better Auth token/session documentation for refresh behavior
