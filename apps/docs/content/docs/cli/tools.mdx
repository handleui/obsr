title: Supported Tools
description: What kinds of signals help Detent extract and fix issues
---

Detent works best when your CI exposes clear failure evidence. Raw logs are valid input. Structured artifacts are optional extra context.

## Best Signals

- File paths
- Line and column numbers
- Tool names
- Rule IDs
- Stack traces
- The exact command that failed

## Common Sources

Detent works well with common CI failure sources such as:

- linters like ESLint, Biome, golangci-lint, and Clippy
- type checkers like TypeScript and Flow
- test runners like Vitest, Jest, Go test, and Cargo test
- build tools and framework failures from Next.js, webpack, Cargo, Docker, and Node.js

## Raw Logs vs. Structured Artifacts

- Raw logs are enough to create issue diagnostics.
- Structured artifacts can improve accuracy when they are easy to produce.
- You do not need to redesign your workflow around a parser-specific file format.

## Setup Tips

- Keep the Detent action as the final `if: always()` step.
- Avoid hiding the real stderr or stdout from failing commands.
- Preserve stack traces when possible.
- Continue to emit reports only if they already fit your workflow.

## What Makes a Good Fix

Detent works best when:

- Errors have file paths and line numbers
- Commands and stack traces are preserved
- The fix is localized (not a refactor)

Complex logic bugs or architectural issues are flagged for human review instead of auto-fixed.
